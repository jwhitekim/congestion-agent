# AGENTS.md

이 문서는 이 프로젝트에서 작업하는 코딩 도구(Claude Code 등)를 위한 설계 지침서다.
코드를 만들거나 고치기 전에 반드시 이 문서의 원칙과 **용어 규칙(0번)**을 따른다.

---

## 0. 용어 규칙 (문서·코드·주석 전체에 강제)

이 프로젝트는 "에이전트 시스템"과 "LLM"을 **명확히 구분**한다. 비유(두뇌/손발)가 실체(LLM/도구)를 대체하지 않게 한다.

| 용어 | 가리키는 것 |
|---|---|
| **에이전트 시스템** | 전체. LLM + 도구 + 도메인 + 실행 루프를 모두 합친 것 |
| **LLM** | 판단을 담당하는 부분 (Claude). 도구 호출을 결정하고 결과를 종합한다 |
| **도구(tool)** | 측정만 하는 부분 (탐지·추적 등). 스스로 판단하지 않는다 |
| **두뇌 (역할 비유)** | LLM의 *역할*을 비유로 부를 때만. 항상 "LLM(두뇌 역할)"처럼 LLM에 붙인다 |
| **손발 (역할 비유)** | 도구의 *역할* 비유. 항상 "도구(손발 역할)"처럼 도구에 붙인다 |

**규칙:**
- **"에이전트"를 단독으로 쓰지 않는다.** 전체를 말하면 "에이전트 시스템", 판단부를 말하면 "LLM"이라고 쓴다.
- **"두뇌"·"손발"을 단독 명사로 쓰지 않는다.** 항상 실체에 붙여 "LLM(두뇌 역할)", "도구(손발 역할)" 형태로만 쓴다.
- 이 규칙은 **코드 주석에도 동일하게 적용**한다. (예: `# LLM(두뇌 역할)이 도구 호출을 결정`, `# 도구(손발 역할)는 사실만 반환`)
- `ClaudeAgent` 클래스는 "LLM + tool use 루프"를 감싼 **에이전트 시스템의 판단 본체**다. LLM 그 자체도 아니고 전체 시스템도 아니다. 주석에서 이 점을 분명히 한다.

---

## 1. 프로젝트 정체성

LLM(두뇌 역할)이 판단하고, CV 등 도구(손발 역할)가 측정하는 **도메인 중립 에이전트 시스템**이다.

구조는 3층으로 분리한다:

- **LLM 층 (`agent/`)** — Claude. 도구 호출을 결정하고, 결과를 종합해 판단하고, 행동을 지시한다. **도메인을 모른다.**
- **도구 층 (`tools/`)** — 객체탐지·추적 등. 시키면 측정만 한다. 스스로 판단하지 않는다.
- **도메인 층 (`domains/`)** — "시스템 프롬프트 + 사용할 도구 묶음"일 뿐이다. 도메인은 여기서만 정의된다.

도메인을 바꿀 때 LLM 층(`agent/`) 코드는 한 줄도 고치지 않는다. 도구와 프롬프트만 교체한다.

---

## 2. 핵심 설계 원칙 (가장 중요)

### 원칙 1 — LLM이 도구를 호출한다 (도구가 먼저 다 돌면 안 됨)

- ❌ 잘못된 구조(그림 A): `영상 → YOLO → OCSort → 밀집도 → (끝나고) LLM 판단`
  도구가 정해진 순서로 무조건 먼저 다 돌고, LLM은 마지막에 도장만 찍는다. 이건 에이전트 시스템이 아니다.
- ✅ 올바른 구조(그림 B): LLM이 프레임을 먼저 보고 → "정보가 필요하다"고 **스스로 판단**하면 → tool use로 도구를 호출 → 결과를 받아 종합 판단.
- 반드시 Claude의 **tool use(function calling) 루프**로 구현한다. 도구를 부를지 말지는 LLM이 정한다.

### 원칙 2 — 도구는 "사실(측정)"만. "판단"은 절대 안 함

- 도구는 "누가 어디에 있고 어떻게 움직이는지"라는 **raw 사실**만 출력한다.
- 도구는 밀집도 점수, 혼잡도 레벨 같은 **해석된 값을 절대 만들지 않는다.** 그건 LLM의 일이다.
- 이유는 아래 "연구 칼날"과 직결된다 — 도구가 밀집도로 뭉개버리면 LLM이 그걸 보정할 수 없다.

---

## 3. 연구 칼날 (이 프로젝트가 풀려는 핵심 문제)

기존 혼잡도 측정의 약점: **카운팅·밀도만 보면 사람의 공간 분포를 반영하지 못해, 한 구석에 몰린 국소 군집을 전체 혼잡으로 오판한다.**

- 예: 한쪽 출구에 8명 몰려 있고 나머지 공간은 비었는데, 밀도 평균만 보면 "혼잡 high"로 오판.
- 해법: LLM(VLM)이 프레임을 보고 **공간 분포를 의미적으로 해석**해 "전체 혼잡"과 "국소 혼잡"을 구분한다.

따라서 **도구는 "어디에 몇 명"이라는 분포 사실까지만** 주고, "그래서 혼잡한가"는 LLM이 분포를 보고 판단한다. 이 경계가 무너지면 프로젝트의 기여 자체가 사라진다.

---

## 4. 도구 정의: `track_people` (탐지 + 추적 묶음)

- 탐지와 추적을 **하나의 도구로 묶는다** (추적은 탐지를 내부에 품으므로 자연스럽다). 동영상이므로 추적은 필수다.
- 내부: YOLOv8m(`yolov8m.pt`, person 클래스) 탐지 → ByteTrack(supervision)으로 ID 부여 및 프레임 간 추적.
- 출력(사실만):
  - 각 사람: `{ track_id, bbox:[x1,y1,x2,y2], center:[cx,cy], frames_present(궤적용) }`
  - 화면을 구역으로 나눈 "구역별 인원 수" 같은 **위치 분포 사실**은 제공해도 좋다.
- **금지:** `밀집도=0.8`, `혼잡도=high` 같은 해석된 점수/레벨 출력.

## 5. 구조 규칙

### 핵심 로직과 데모(웹)를 분리한다

`agent_system/`은 순수 에이전트 시스템이다. 웹을 몰라도 된다. 웹/프론트는 그걸 바깥에서 보여주는 껍데기이므로 분리한다.

```
LLMAgentSystem/
  AGENTS.md            ← 이 문서
  README.md
  requirements.txt
  agent_system/        ← 순수 에이전트 시스템 (웹 모름)
    agent/             ← LLM 층 (도메인 중립). ClaudeAgent = 판단 본체
    tools/             ← 도구 층 (base.py = 공통 인터페이스)
      congestion/
    domains/           ← 프롬프트 + 도구 묶음
    utils/
    main.py            ← CLI 진입점
  web/                 ← 데모용 웹 (agent_system을 import해서 씀)
    web.py             ← 서버 (Flask/FastAPI)
    frontend/
      index.html
      app.js
      styles.css
  logs/
```

- `web.py`는 `agent_system/` **밖으로** 뺀다. 에이전트 시스템은 CLI(`main.py`)·웹·배치 어디서든 import해 쓰일 수 있어야 한다.
- 프론트엔드는 **React 안 씀.** 데모 시각화용이므로 순수 HTML/CSS/JS로 충분하다.

### LLM 층(`ClaudeAgent`) 클래스 규칙

- 클래스 이름이나 내부에 `congestion` 같은 **도메인 단어를 박지 않는다.** (예: `CongestionAnalyzer` ❌ → `ClaudeAgent` ✅)
- `ClaudeAgent`는 `system_prompt`와 `tools`를 **주입받는다.** 도메인 로직을 하드코딩하지 않는다.
- `ClaudeAgent`는 "LLM + tool use 루프"를 감싼 **판단 본체**다. 주석에서 LLM 그 자체나 전체 시스템과 혼동하지 않게 쓴다.

## 6. 도메인 프롬프트 (congestion 예시)

시스템 프롬프트 취지:

> 너는 혼잡 상황을 판단하는 LLM이다(두뇌 역할). `track_people` 도구(손발 역할)는 사람들의 위치와 움직임이라는 '사실'만 준다.
> 너의 일은 그 사실을 해석하는 것이다. 특히 사람이 한 구역에 몰려 있는지, 고르게 퍼져 있는지(공간 분포)를 보고
> '전체가 혼잡한지'와 '특정 구역만 국소적으로 혼잡한지'를 구분해서 판단하라.
> 단순히 사람 수가 많다고 혼잡으로 판단하지 마라 — 분포를 봐라.
> 필요하다고 판단되면 도구를 호출하고, 최종 답은 아래 JSON으로만 하라:
> `{ total_people, distribution_summary, congestion_level(low/medium/high), local_hotspots, reasoning(한국어), action(none/monitor/alert) }`

## 7. 서랍에 넣어둔 카드 (지금은 넣지 말 것 — 필요가 증명되면 꺼냄)

아래는 "있으면 좋을 것 같아서" 미리 넣지 않는다. 박스 기반으로 먼저 돌려보고, 실제로 한계가 확인된 다음에 도입을 검토한다.

- **세그멘테이션** — 가림(occlusion) 상황에서 부분적으로 가린 사람을 픽셀로 떼어내는 데 유리. V100 32GB×2 있어 VRAM 부담은 없음. 단, 가림이 실제 문제로 확인되고 LLM의 정황 추론으로도 안 될 때 도입.
- **자세 추정 / 행동 분석** — 위치(탐지)와 다른 "새로운 축"(정적 자세 / 동적 동작)이 판단에 실제로 필요해질 때. 같은 축을 두 번 보는 도구는 오히려 LLM을 혼란시킨다.
- **멀티뷰(다른 각도 영상)** — 완전 가림(한 카메라에서 100% 숨음)의 근본 해법.
- **도메인 후보: 매장 좌석 점유** — "자리비움(화장실 등) vs 진짜 빈자리", "테이크아웃 vs 머무는 손님" 구분. 시간/의도 판단이 필요해 LLM의 명분이 강함. 유력하나 미확정.

도구를 고를 때 기준: **"서로 다른 축을 보는 도구"라야 LLM에 도움이 된다.** 같은 질문(위치)에 답하는 도구를 늘리면 상충만 늘어난다.

- [ ] `web.py`를 `agent_system/` 안에 두는 것

## 9. 기술 스택

- Python, anthropic SDK, ultralytics(YOLOv8), opencv-python, ByteTrack(supervision)
- 데모 웹: Flask 또는 FastAPI + 순수 HTML/CSS/JS
- 하드웨어: V100 32GB ×2 (주), TITAN X Pascal 12GB
- 주석은 한국어로. 0번 용어 규칙을 지켜서 "여기서 LLM(두뇌 역할)이 도구 호출을 결정", "도구(손발 역할)는 사실만 반환", "여기서 LLM이 분포를 해석" 같은 구조 설명 주석을 단다.
